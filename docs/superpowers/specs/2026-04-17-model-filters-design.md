# Быстрые фильтры моделей — дизайн

## Контекст

Текущая страница `frontend/src/app/models/page.tsx` уже получила первую версию быстрых фильтров, но пользователь уточнил UX:
- фильтры должны быть не вертикальными группами, а одним компактным облаком чипов
- без отдельных заголовков групп
- при выборе нескольких условий модель должна совпадать со **всеми** выбранными условиями, а не с любым из них
- нужна сортировка по:
  - дате добавления модели на сайт
  - количеству загрузок
  - количеству лайков

Важно: первая версия по-прежнему не должна включать миграции БД и не должна превращаться в большой рефакторинг страницы.

## Цель

Сделать на странице каталога моделей компактный блок фильтров, где:
- `Status` остаётся single-select
- дополнительные фильтры выглядят как единое облако чипов
- логика выбора соответствует ожиданию пользователя: если выбрано несколько условий, модель должна проходить по всем
- есть сортировка по популярности и новизне

## Рекомендуемый UX

Используется единый компактный блок:
1. Search
2. Filter cloud
3. Sort by
4. Clear filters

Без визуального деления на `Size / Quant / Tags` заголовками.

## Набор фильтров

### Status chips
Оставить:
- `All`
- `Active`
- `Pending`
- `Deploying`
- `Inactive`

Поведение:
- single-select
- работает как отдельная ось фильтрации

### Property / tag chips в одном облаке

Оставить в одном общем облаке:
- `<10B`
- `10–30B`
- `30–70B`
- `70B+`
- `FP16`
- `Q8`
- `Q4`
- `GGUF`
- `Uncensored`
- `Qwen`
- `Gemma`
- `Reasoning`
- `Abliterated`
- `MoE`
- `Heretic`
- `GPT-OSS`

## Поведение фильтров

### Status
- single-select
- только одно значение одновременно

### Size
- single-select
- только один диапазон одновременно
- если не выбран, не участвует в фильтрации

Причина: size-buckets взаимоисключающие, AND между ними почти всегда даст пустую выдачу.

### Quant
- single-select
- только один quant/filter формат одновременно
- если не выбран, не участвует в фильтрации

Причина: пользователь хочет семантику “выбрал два — должны совпасть оба”, но для quant это даёт нерабочий UX (`FP16` AND `Q4` почти всегда пусто). Поэтому quant должен быть взаимоисключающим переключателем, как size.

### Derived tags
- multi-select
- логика между выбранными tags: **AND**

Пример:
- `Qwen` + `Reasoning`

Показывать только модели, которые одновременно определяются как `Qwen` и `Reasoning`.

## Итоговая логика комбинирования

Финальный фильтр:
- `search` AND `status` AND `size` AND `quant` AND `everySelectedTag`

То есть:
- text search — обычный contains
- status — single-select
- size — single-select
- quant — single-select
- selected tags — AND между всеми выбранными tag chips

Пример:
- `active`
- `30–70B`
- `Q4`
- `Qwen`
- `Reasoning`

Результат:
- только active модели
- только 30–70B
- только Q4
- и одновременно `Qwen`
- и одновременно `Reasoning`

## Сортировка

Добавить отдельный dropdown `Sort by` со значениями:
- `Newest added`
- `Most downloads`
- `Most likes`

### Порядок применения
Сначала фильтрация, потом сортировка.

### Источники данных
- `Newest added` → `created_at`
- `Most downloads` → `hf_downloads`
- `Most likes` → `hf_likes`

## Требование к данным

Чтобы сортировка `Newest added` работала, frontend-модель должна иметь `created_at`.

Это означает:
- если поле уже приходит из backend, нужно просто добавить его в frontend interface и использовать
- если поле не приходит, нужно расширить frontend API payload так, чтобы `created_at` приходил в список моделей

Важно:
- это допустимое изменение API/schema response
- миграции БД для этого не нужны, поле уже существует в таблице `llm_models`

## UI-поведение

### Layout
- Search остаётся сверху
- ниже единый `filter cloud`
- ниже/рядом dropdown `Sort by`
- ниже кнопка `Clear filters`
- без заголовков групп `Size`, `Quant`, `Tags`

### Visual style
- использовать те же chip/button стили, что уже применяются к текущим фильтрам
- активные чипы должны быть визуально заметны
- status chips и остальные chips могут жить в одном блоке, но status должен оставаться отдельной логикой

### Result summary
Показывать короткую строку вида:
- `23 models found`
или
- `23 models • sorted by downloads`

Первая версия может ограничиться кратким summary без полной расшифровки активных фильтров.

## Clear behavior

Кнопка `Clear filters`:
- сбрасывает `status` → `All`
- сбрасывает `size`
- сбрасывает `quant`
- сбрасывает все selected tags
- сбрасывает sort → `Newest added`
- **не** сбрасывает search

Причина: search пользователь, как правило, воспринимает отдельно от toggle-фильтров.

## Public / non-admin поведение

Для public/non-admin нельзя полагаться на урезанный `/api/v1/models` payload для derived filters и sorting по дате добавления, если там нет нужных полей.

Поэтому допустимы два варианта, из которых рекомендуется первый:

### Рекомендуемый вариант
- если metadata для public недостаточна, filter cloud и advanced sorting показываются только admin-пользователю
- public остаётся на безопасном `search + active-only list`

### Альтернатива
- подтянуть недостающие поля в public payload и дать тот же UI всем

Для первой итерации рекомендуется не ломать public path и не делать лишних backend-изменений без необходимости.

## Реализация

### Первая версия
Можно модифицировать:
- `frontend/src/app/models/page.tsx`
- при необходимости `frontend/src/lib/api.ts`
- при необходимости backend response schema / router только в части отдачи `created_at`, если его сейчас нет в frontend payload

### Не делать
- миграции БД
- серверную фильтрацию
- сохранение состояния в URL
- отдельную боковую панель
- drag/drop/сложную композицию фильтров

## Тестирование

Локальные тесты пользователь не хочет. Проверка делается после деплоя через UI.

Минимальный чек-лист:
1. Status остаётся single-select
2. Size остаётся single-select
3. Quant остаётся single-select
4. Tags работают как AND
5. Поиск комбинируется с фильтрами
6. Sort by меняет порядок уже отфильтрованного списка
7. `Newest added` использует реальные новые модели, а не случайный порядок
8. `Clear filters` сбрасывает всё кроме search
9. Public/non-admin path не ломается

## Границы и риски

### Допустимые ограничения
- derived tags остаются эвристикой, а не canonical metadata
- отдельные false positive / false negative по тегам возможны
- без реальных HF tags в БД это допустимая компромиссная первая версия

### Почему это приемлемо
Пользователь просит быстро довести UX до удобного рабочего состояния. Единое облако фильтров + сортировка дают заметный прирост usability без тяжёлого backend проекта.
